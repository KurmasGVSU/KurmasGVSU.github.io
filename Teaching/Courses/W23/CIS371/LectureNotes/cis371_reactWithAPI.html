Color List / React with API

Props Down / State Up

  • Demonstrate ColorList
  • Sketch App -> ColorList -> Color -> StarRating -> Star
  • Where should the numerical ranking be stored?
    • In general, as “low” as possible. In this case it could be the StarRating.
    • However, because we need to send changes to the API, this data is actually stored in the App component.
  • When data is stored “above” the component with the listener, we need a way of calling the appropriate function in the parent component.
    • In other words, the Star’s onClick property needs to be set to a function inside App.
  • We write a function updateRating inside App.
    • This function takes as input the star’s ID and new rating,
    • then updates the state accordingly (using the set function returned by the useState hook)
  • We must now pass this function down through the hierarchy.
    • Notice that ColorList has an update attribute.
    • Notice also that the value of this attribute is the function object. We are not invoking the function, just passing it along.
  • Color has an attribute onColorRatingUpdated
  • StarRating has an attribute onRatingUpdated
  • In principle, this function object is being passed down through the hierarchy.
  • When the item at the bottom of the hierarchy is clicked, the function that was passed down executes.
    • This is similar to the JavaScript MVC setup where the controller hands the callback to the View.
  • Imagine the President writing down some instructions, putting them in an envelope, and handing them to the Provost.
    • The Provost passes the envelope to the Dean
    • The Dean passes the envelope to the Director
    • The Director passes it to me,
    • Then I open the envelope and follow the directions.
    • When you write a function’s name only, you are “passing the envelope”
    • When you write a function, followed by parentheses (), you are opening the envelope and following the directions.
  • Now, the story is a bit over-simplified in this case, because we need to stop on the way up and add some data.
    • Star has the attribute onStarClicked. The value passed here is simply passed to the FaStar component.
  • When we click on a star, we need to know which star was clicked (so we know the rating). Thus, StarRating passes a lambda to the star’s onStarClicked. This lambda adds the star’s numeric value when calling the “upstream” lambda.
  • Likewise ColorList adds the color’s id when passing the call “up”.

API Call

  • Now we want to grab data the “real” way: From an API server
  • show ColorListAPI
    • Very simple because it doesn’t have any authentication/login
    • Notice use of port 3001. (API and React need to use different ports)
    • Notice the use of res.json to send raw data
    • visit localhost:3001/colors and see raw .json returned.
  • fetch makes an network request and returns a promise with the response.
    • request is GET by default, but this can be configured.
    • Notice that handling .json is a two-step process. First then gets the response as text. Next, we need to parse it as JSON to create the desired JavaScript object.
    • Once we have the object, we can use the state hook to set the colors.
  • useEffect hook is code that is (potentially) run after every render (i.e., after every time components are redrawn — which happens every time state changes.)
    • useEffect calls fetchColors.
      • Notice the [] as the 2nd parameter. This is the list of objects the hook should watch, and only re-run when items in the list change.
      • We only want to fetch colors upon page load, so we pass an empty array.
      • See what happens if the brackets are removed.

CORS

  • Remove CORS headers from the API. Then try to run the app. Notice the CORS error in the browser.
  • The problem is that JavaScript sent from website X (the React server) is trying to connect to website Y (the ColorsAPI)
  • Why is this a security issue? (Hint: Think about cookies.)
    • Suppose you are logged into Facebook on Tab A. Then you visit kittenpics.com in Tab B; but that site contains JavaScript code that makes a fetch call to Facebook. Your browser will send the session cookie and return the private data.
  • Why is this a security risk? Your data is being sent to _your_machine?
    • Because the “bad” JavaScript could take the data and send it somewhere else.
  • Add the CORS header: res.setHeader("Access-Control-Allow-Origin", "http://localhost:3000")
    • The protocol, hostname, and port must all match.
  • Who is responsible for implementing this security feature? The browser or the server?
    • The browser. The server returns the data, but the browser dumps it when it doesn’t find the correct CORS header.
    • The security risk isn’t in sending the data (the credentials are, present after all). The security risk is in what the browser does with it.
    • You can configure some browsers to ignore CORS issues.
    • You can write your own browsers to handle data however you want.
  • Note: This is just one small part of CORS. It gets more complicated. Sorry.

Post

  • Turn CORS back on. Go to ColorList2. Add a color.
  • Look at NewColorForm.jsx.
    • Text field
    • Color picker
    • Two buttons
  • Notice the form has an onSubmit property.
  • There are two ways to “connect” HTML input items to React
    • Controlled, and
    • Uncontrolled.
  • With Uncontrolled, The HTML component stores it’s own state, and React just grabs the current state when the form is submitted.
  • With controlled, we use React to store the component’s state, so React always has the up-to-date values.
  • ColorList2 uses controlled components: Every time a field is change, App#updateFormData is called and updates a state hook containing the data for the Color object currently being edited.
  • When the form is submitted, App#submit is called.
    • Need to call event.preventDefault to prevent the default behavior for forms, which is sending the form data to a URL and loading a new page.
    • Don’t forget this step
    • fetch is called and configured to send a POST request.
      • Notice that parameters are all in an object. (This makes them effectively keyword parameters.)
      • Notice the content type (json)
  • API sends new Color object back.
    • Why?
    • DB creates a PK. Client side needs to know that new PK.
  • There is another level of CORS here.

CORS, part 2

  • When fetching all the colors, the server sent the data and the browser kept it out of the hands of the calling script.
  • This scenario doesn’t work for POST.
  • Why?
    • What if the POST request was “Pay Bob $10,000?”
    • Because the POST can potentially change the DB state, we need to be sure it was actually authorized/intended by the user.
    • Remember: If the user is logged in using another tab, the session cookie will be sent.
  • As a result, the browser need to do a “preflight”: It makes a request using the verb OPTIONS to make sure the cross-origin request is allowed.
  • When making cross-origin requests, many “non-standard” parameters need to be specifically authorized. In this case, the use of the json content-type. Hence, the additional line in the use block.
  • To tighten things up, there are also CORS headers to allow/disallow things like specific verbs (GET/POST/DELETE) and/or headers.

Update

  • Notice that we use the same form for both adding a color and updating color.
  • The editMode property tells the form whether to set up for an “Add” or an “Update”
    • It controls the labels.
  • Each color has an edit button. Pushing the edit button changes the editMode state in App, which re-draws the form.
    • It also sets the colorToEdit state, so that color info is displayed in the form.
