Semaphores

  • A semaphore combines locks and condition variables.
  • Invented by Dijkstra (of Dijkstra’s algorithm and “goto considered harmful”)
  • A semaphore is an object with an integer value.
  • sem_wait:
    • decrement the integer.
    • wait if value is negative
    • historically called P
  • sem_post:
    • increment the integer
    • wake one waiting thread.
    • historically called ‘V’

Binary semaphores (locks)

  • Initialize semaphore to 1.
  • First to call wait decrements to 0 and moves on.
  • If second comes along, decrements to -1 so waits.
  • When first releases, count goes to 0 and second thread wakes.
  • If first finishes before second comes along, semaphore returns to 1 allowing second to enter immediately.
  • How to use semaphore to wait for child?
    • Binary semaphore set to 0.

Bounded Buffer

  • Use two semaphores, one for empty and one for full.
  • Works for one producer and one consumer.
  • Fails for multiple producers and consumers. Why?
    • Can’t fill/empty at the same time. There’s a race condition.
  • Solution: Need mutex.

    • Deadlock

Reader/Writer locks

  • Imagine a database that is read and written my many different processes.
  • What is interesting about getting the locking write for reading and writing?
    • Multiple reads allowed, but only one writer.
    • Writing also requires readers to be locked out.
      • Why?
  • So, what do we do?
    • Basic mutex for writing (can use a semaphore).
    • Count readers
    • First reader also obtains writelock, preventing writing.
    • Last reader releases writelock allowing writing again.
  • What’s the problem/limitation?
    • Writers can starve.
    • Two step process adds overhead. Make sure that reader/writer lock is actually faster than simply doing mutually exclusive reading. (Hill’s Law)
