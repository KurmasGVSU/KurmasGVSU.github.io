Class Design / Intro to UML

  • Imagine you want to design a class to describe a savings account
    • What properties (instance variables) would it have?
      • name, address, balance, interest rate
    • What instance methods would it have?
      • deposit, withdraw, payInterest, printStatement
  • This is true regardless of the language you use (Java, Python, C++, etc.)
  • UML stands for _U_nified _M_arkup _L_anguage
  • It is a language-independent way of describing how a program is built.
    • (Think of it kind of like a blueprint)
  • Public items listed with “+”
  • Private items listed with “-“

  • Like python, the types go after the names.

  • Write Savings Account.

Inheritance

  • Suppose you are writing software for a bank and they also want to support Checking Accounts.
  • How do they differ:
    • SavingsAccount only could have payInterest
    • CheckingAccount could have cashCheck
    • Both have a printStatement
  • Notice that both classes are nearly identical. They share some code, and are unique in some ways also.
  • Because of the DRY principle (and general laziness), we want to avoid duplicating code.
  • The principal of inheritance allows us to create a class by extending and/or modify the behavior of another.
  • Refactor the common code into a BankAccount class.
  • class CheckingAccount extends BankAccount
    • CheckingAccount behaves as if it contains all of the BankAccount code. (Imagine that you cut-and-paste all of the BankAccount code into CheckingAccount)
  • You can modify behavior by
    • Adding additional methods (e.g., cashCheck)
    • Replacing methods (e.g., printStatement)
      • This is called overriding
  • If you need to refer to the parent class, use super
    • e.g., super(foo, bar) to invoke parent constructor.
    • e.g., super.printStatement to print the user’s information.
