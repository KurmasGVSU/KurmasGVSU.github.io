Complete VM Systems.

VAX/VMS

  • VAX was from DEC.
  • VMS OS needed to run on a wide variety of hardware.
  • 32-bit address space, 512 byte pages
    • (Yes, that’s small)
  • Address space uses two bits for segment:
    • “bottom” half is “process space”
    • Code and heap that grows down
    • Stack that grows up.
    • “top” half is for OS.
  • Page tables live in kernel space, and can be swapped out.
  • Translating virtual address is complicated
    • Consult system page table to get page containing process page table.
  • VAX had hardware-managed TLBs
  • Page 0 is invalid to simplify null-pointer debugging.
  • Sharing the kernel segment with all processes simplifies things like copying data to/from OS buffers to processes.
  • Each page has protection bits
  • No reference bit
  • Second chance list:
    • If a process uses too many pages, one is chosen from a FIFO queue and added to either a global clean or dirty list.
    • If the original process requests that page before it is evicted, it is “saved” and no need to read from disk.
  • Pages swapped in and out it clusters.
  • Demand zeroing.
  • Copy on write
    • When page copied (e.g., through fork()), only page table entry made.
    • Page only copied if one process tries to write.

Linux

  • Page 0 is invalid
  • Kernel space is roughly upper 25% of vm space and is shared among processes (Much like vax)
  • Kernel has kernel logical addresses which map more-or less directly to physical addresses
    • 0xC0000000 -> 0x00000000
    • Contiguous kernel logical addresses are contiguous physical addresses.
    • Important for things like virtual memory.
  • Kernel virtual addresses mapped in a more traditional way
  • Useful for large buffers where contiguous memory would be hard to find.
  • x86 provides hardware-managed, multi-level page table with one table per process.
    • On context switch point privileged register at page table entry. Hardware MMU does the rest.
  • 64-bit x86 uses a four-level page table structure.
  • Only lower 48 bits used currently
    • 64 Petabytes
  • Lowest 12 bits offset into a page.
  • Middle 36 bits need to be translated: 9 bits per level.
  • As more bits needed, page table will eventually be six levels.
  • Linux supports “huge pages” (2MB, 1GB)
    • Main motivation: Better TLB behavior
    • Also shorter TLB miss path.
    • TLB misses can be up to 10% in some workloads.
  • Linux uses the same system to cache pages in memory for all three types of disk data:
    • Memory mapped files
    • files accessed through read and write
    • Memory pages from stack and heap.

2Q Replacement Policy

  • LRU is good, but has some common pathological behavior (i.e., chasing tail through large array).
  • 2Q maintains two lists: active and inactive.
    • page promoted to active upon 2nd reference.
    • pages evicted from inactive list.
    • pages occasionally moved form active to inactive to keep balance approximately 1/3 to 2/3.
  • Each list uses the clock algorithm for reasons discussed earlier.

Security

  • To cut down on buffer overflow, No Execute bit: Can’t treat data on stack as code.
  • Return oriented programming; Manipulate the stack so that stack of return addresses cause desired code to run.
  • ASLR: Address Space Layout Randomization
    • If you print address of stack variable, it will be different every time.
