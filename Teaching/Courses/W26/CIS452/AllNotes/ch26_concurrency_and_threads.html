Concurrency and Threads

  • Threads – Two views:
    1. A process with multiple Program Counters each of which is being fetched and executed from
    2. A bunch of separate processes that share the same address space.
  • How is switching threads similar to switching processes? How is it different?
    • Similar: Need to preserve PC and registers.
    • Different: No need to change page table.
  • Advantages and disadvantages of threads vs. processes:
    • Threads:
      • No cache/TLB effects of changing address space
      • Lower overhead of context switch
      • Easier communication (no need to involve OS)
      • You don’t have to give up the CPU if one thread blocks on I/O – just transfer to another thread.
      • Not obvious how to manage stack — need to allocate a fixed size.
    • Processes:
      • Isolation: A failure in one doesn’t bring down all.
  • Ordering controlled by scheduler.
  • Common situations
    • Create A
    • Create B
    • Run A
    • Run B
  • Or
    • Create A
    • Run A
    • Create B
    • Run B.
  • But, the following is possible (but unlikely)
    • Create A
    • Create B
    • Run B
    • Run A

  • Race Condition: A condition where the outcode depends on the timing of the code’s execution
    • From Claude: “A race condition is the window of time between when a thread reads shared state and when it writes it back — during which another thread can invalidate the read value.”
  • Critical section: A piece of code with a potential race condition.
  • General solution is to make operations atomic.
  • But, can’t implement every imaginable atomic operation in HW
  • Instead, need a few key primitives in HW in Operating system that provides reasonably efficient mutual exclusion when necessary.

  • Efficiency requires threads be blockable and wakeable (Requires OS support.)
