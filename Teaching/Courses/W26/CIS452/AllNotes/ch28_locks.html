Building a lock

  • Need help from HW and OS.
  • what are key criteria?
    1. Mutual exclusion
    2. Fairness (nobody starves)
    3. Performance / Overhead
  • Simple solution: Disable interrupts during critical section
    • Requires a lot of trust
    • could work in very limited situations
    • Bugs can bring down the entire machine.
    • If interrupts disabled too long, other problems arise
    • Only works on single CPUs. (Why?)

  • Attempt 2: Spin-wait

    typedef struct __lock_t { int flag; } lock_t;
  
void init(lock_t *mutex) {
 // 0 -> lock is available, 1 -> held
 mutex->flag = 0;
}
  
void lock(lock_t *mutex) {
  while (mutex->flag == 1) // TEST the flag
     ; // spin-wait (do nothing)
  mutex->flag = 1; // now SET it!
}
  
void unlock(lock_t *mutex) {
  mutex->flag = 0;
}
  • Is this correct? Why or why not?
    • What happens if multiple threads are waiting and see mutex->flag != 1?
  • Is is this performant?
    • No. Spin wait is wasted time.
  • What is the consequence of spin waiting on a single CPU?
    • Effectively no concurrency during critical section.
  • What fundamental operation do we need to fix the correctness issue?
    • Atomic test-and-set (xchg “atomic exchange” on x86)
    int TestAndSet(int *old_ptr, int new) {
  int old = *old_ptr; // fetch old value at old_ptr
  *old_ptr = new; // store ’new’ into old_ptr
  return old; // return the old value
}

    ; int TestAndSet(int *old_ptr, int new)
; args: rdi = old_ptr, esi = new
TestAndSet:
    mov eax, esi        ; eax = new
    xchg eax, [rdi]     ; atomically swap eax with *old_ptr
                        ; eax now holds old value, *old_ptr = new
    ret                 ; return old value in eax
  • Notice what’s going on here:
    • If you want a lock, pass a 1 as new.
    • Lock will be set upon completion.
      • Either already set, or will now be set.
    • Returning old will tell you if the lock was previously set, or if it was not set (and is now yours).
  • Important: This only works if the function is atomic.
    • C code is just for illustrative purposes.

  • Possible lock implementation using TestAndSet:

    void lock(lock_t *lock) {
  while (TestAndSet(&lock->flag, 1) == 1)
     ; // spin-wait (do nothing)
}
  • This is correct.
  • Is it fair? Why or why not?
    • Not fair. Starvation is possible.
  • What about performance
    • Terrible on single CPU.
    • Not too bad when number of cores is similar to number of threads if critical section is very small
  • Why is it important that the number of threads be <= number of cores?

Peterson’s algorithm.

  • It is possible in specific circumstances to implement a spin lock without special hardware:

    ~~~ int flag[2]; int turn;

    void init() { // indicate you intend to hold the lock w/ ’flag’ flag[0] = flag[1] = 0; // whose turn is it? (thread 0 or 1) turn = 0; }

    void lock() { // ’self’ is the thread ID of caller flag[self] = 1; // make it other thread’s turn turn = 1 - self; while ((flag[1-self] == 1) && (turn == 1 - self)) ; // spin-wait while it’s not your turn }

    void unlock() { // simply undo your intent flag[self] = 0; } ~~~

  • Why is there no race condition?
    • Separate flags for each process.
  • Why do we need both flag and turn? (Why is flag not sufficient?)
    • flag alone can lead to deadlock.
    • setting turn to the other process ensures progress.
    • turn alone is not sufficient, because you could get stuck (i.e., violate progress)
  • Compare and Swap
    • Similar, but more powerful than test and set.

    • (We’ll use the “extra” power later)

      int CompareAndSwap(int *ptr, int expected, int new) {
  int original = *ptr;
  if (original == expected)
      *ptr = new;
  return original;
}
  • Load linked and store conditional
    • Used by Alpha, PowerPC, ARM, and MIPS
    • load linked is a typical load

    • store conditional only stores if no other stores have taken place.

      void lock(lock_t *lock) {
  while (1) {
    while (LoadLinked(&lock->flag) == 1)
         ; // spin until it’s zero
    if (StoreConditional(&lock->flag, 1) == 1)
        return; // if set-to-1 was success: done
    // otherwise: try again
  }
}

      Alternatively,

      void lock(lock_t *lock) {
   while (LoadLinked(&lock->flag) || !StoreConditional(&lock->flag, 1))
      ; // spin
}

  • Fetch-And-Add

    int FetchAndAdd(int *ptr) {
  int old = *ptr;
  *ptr = old + 1;
  return old;
}
    • Return value is the thread’s “turn”
    • Guarantees progress
    typedef struct __lock_t {
  int ticket;
  int turn;
  } lock_t;
   
  void lock_init(lock_t *lock) {
  lock->ticket = 0;
  lock->turn = 0;
  }
   
  void lock(lock_t *lock) {
  int myturn = FetchAndAdd(&lock->ticket);
  while (lock->turn != myturn)
  ; // spin
  }
   
  void unlock(lock_t *lock) {
  lock->turn = lock->turn + 1;
  }

Too much spinning

  • Notice that the solutions above do not require OS support – just the atomic HW primitive.
  • But we can waste entire time slices
    • when there is contention, and
    • especially when the lock is held for a long time.
  • Simple OS solution: Call yield when you begin to spin.
  • What are the limitations of this approach?
    • If there are many threads, and one gets preempted in a critical section, each other thread will likely get scheduled only to yield. This is a lot of context switches for no progress.
    • It makes starvation more likely: While a process is off the CPU, another can come in and grab the lock.
  • Solution:
    • Add a mechanism to block and unblock processes.
    • Maintain a queue
  • If a process begins to spin, add it to a queue, then
  • unblock in order.
  • Notice that there is still a spin lock, but it contains only a few instructions.

  • Probably of time slice expiring while in critical section is small.

    ~~~ typedef struct __lock_t { int flag; int guard; queue_t *q; } lock_t;

    void lock_init(lock_t *m) { m->flag = 0; m->guard = 0; queue_init(m->q); }

    void lock(lock_t *m) { while (TestAndSet(&m->guard, 1) == 1) ; //acquire guard lock by spinning if (m->flag == 0) { m->flag = 1; // lock is acquired m->guard = 0; } else { queue_add(m->q, gettid()); m->guard = 0; park(); } }

    void unlock(lock_t *m) { while (TestAndSet(&m->guard, 1) == 1) ; //acquire guard lock by spinning if (queue_empty(m->q)) m->flag = 0; // let go of lock; no one wants it else unpark(queue_remove(m->q)); // hold lock (for next thread!) m->guard = 0; } ~~~

  • Notice the use of a spin lock “guard” to protect code that sets up the “real” lock
    • This should be fine because it is only held for a few lines of code.
    • (“Fancier” approaches would require more cycles than just spinning.)
  • What happens if park is called before guard is released?
  • When process waiting for a lock, flag is not set back to 1. Why?
    • Lock is handed directly to next process in line.
  • Do you see any subtle race conditions here?
  • Think about the interaction between checking the queue and parking/unparking?
    1. Process A is running lock, adds itself to the queue, sets guard to 0, then gets taken off the CPU.
    2. Process B calls unlock, sees queue is not empty, signals Process A by calling unpark.
    3. Process A gets scheduled again. It’s next like of code is park. However, the last process holding the lock already called unpark(A). When A calls park, it will go to sleep and not wake up because nobody knows to signal it.
  • Solution:
    • Call setPark() before releasing the guard. If there is an intervening unpark then the system can handle it.
    • Build an atomic kernel call to release guard and park.
