HTTPS

Background

  • Two important aspects to secure communication:
    1. Others can’t read the message
    2. Others can’t change the message.
  • Two main approaches:
    • Symmetric (shared) key
    • Asymmetric (public/private) key
  • To ensure only recipient can read message:
    • Encode with his public key.
    • Private key required to make sense of message.
  • To ensure message comes from known source:
    • Encode with sender’s private key.
    • Recipient must decode with sender’s public key.
    • Message must have come from sender — otherwise it would be gibberish
  • Symmetric key is faster, but
    • What’s the problem?
    • How to share the key
  • Idea: Use public/private once to share the key then switch to symmetric
    • Client requests secret key from server
    • Server encrypts secret key using its own private key and sends it to the client.
    • Client decrypts secret key using server’s public key.
  • What’s the problem?
    • How does the client get the server’s public key?
    • Can’t trust regular HTTP.
    • Could go to a physical location (e.g., the bank) and pick it up.
    • (Notice this is still not perfect — employee could be corrupt. Humans are the weak point of all systems.)
  • Impractical to physically collect public keys.
    • How to make it scale?
