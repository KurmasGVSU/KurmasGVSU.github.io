I/O

  • What do you expect the main factors in bus performance are?
    • Length
    • Number of attached items
    • Traffic
    • Component quality
  • Busses tend to form a hierarchy
  • High-performance busses (e.g., memory) tend to be
    • short and
    • expensive
    • plus less contention improves performance.

  • When performance is less critical, we can use slower, longer, busses with more stuff attached.

  • Intel example
    • Dedicated memory bus
    • Dedicated bus for graphics
    • Other I/O goes through an external chip using DMI.
    • Various busses/protocols attach to I/O chip (AT, SATA, eSATA, USB, PCIe, etc.)
  • Common simple I/O Device: Three registers
    • Status
    • Command
    • Data
  • When the main CPU communicates directly with the device, it is programmed I/O
    • Programmed IO addresses devices through Ports
  • Polling is inefficient. Better idea: Interrupts
    • Put process to sleep,
    • wake it up when the device issues an interrupt.
    • (work best for slow devices)
  • Direct Memory Access (DMA)
    • Programmed I/O takes up a lot of CPU time with trivial tasks
  • Ports vs. Memory Mapped I/O
    • with MMIO, device ports are assigned memory addresses
    • I/O is now effectively just load/store.
